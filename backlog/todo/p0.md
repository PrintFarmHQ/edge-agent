# PrintFarm Edge — Bambu Cloud Integration Plan (4 Phases, TDD-first)

## Goal
Add production-grade Bambu support to the Go edge agent while preserving the existing control-plane contract.

## Delivery Sequence (Requested)
1. Phase 1: Authenticate to Bambu Cloud (including MFA when required)
2. Phase 2: Retrieve cloud printer list and build local printer state in edge-agent
3. Phase 3: Send print command to any available printer
4. Phase 4: Implement remaining Bambu commands and full lifecycle behavior

## Locked Decisions (Agreed)
- Bambu mode is cloud-first.
- Bambu discovery is cloud enumeration primary, with LAN/Bambu Connect fallback when cloud auth/API is unavailable.
- Control-plane contract stays unchanged: keep `bindings`, `desired-state`, and `state-push` model.
- Existing flags and existing functionalities are strict compatibility constraints and must not change in any phase.
- Adoption ownership is in SaaS, not edge.
- Edge does not keep adopted/not-adopted truth for printers.
- Edge runs as a long-lived process started via flags (no standalone login workflow).
- Bambu startup auth requires:
  - `--bambu`
  - `--bambu-username`
  - `--bambu-password`
- `--bambu-password` is accepted as-is.
- Passwords are never persisted; token material is persisted.
- Bambu auth failure degrades Bambu path only; agent continues for other capabilities.
- Local state path for this work: `~/.printfarm-edge/`.
- Existing ecosystem packages can be used (including sidecar approach if required).
- No reverse-engineering activities are part of this plan.

## Non-goals
- AMS inventory management
- Camera streaming
- Slicing in edge-agent
- Broad control-plane schema redesign

---

## Runtime Invocation Contract
1. Keep edge-agent as a long-lived process launched directly with flags.
2. Preserve operator model:
   - `./edge-agent --api-key=<saas_key> --klipper`
   - `./edge-agent --api-key=<saas_key> --klipper --bambu --bambu-username=<username> --bambu-password=<password>`
3. Preserve all existing flags with identical names, parsing rules, and behavior.
4. Validate at startup that `--bambu` requires both `--bambu-username` and `--bambu-password`.
5. Do not add standalone `bambu login/logout/status` workflows.
6. MFA handling is startup-driven:
   - optional `--bambu-mfa-code=<code>`
   - optional `--bambu-mfa-code-cmd=<command>` to fetch a one-time code at startup
   - if MFA is required and no code source is provided, Bambu initialization fails with explicit MFA-required error while agent keeps running.

- Persist token metadata and refresh material in local secure store.
- Support MFA challenge flow when cloud requires it.

### Inputs
- Required when `--bambu`:
  - `--bambu-username`
  - `--bambu-password`
- Optional MFA inputs:
  - `--bambu-mfa-code`
  - `--bambu-mfa-code-cmd`

### Persistence
- `~/.printfarm-edge/bambu/credentials.json` (`0600`)
- fields:
  - `access_token`
  - `refresh_token` (if available)

## Phase 2 — Cloud Printer List + Local State

### Scope
- Retrieve printers from cloud API using Phase 1 auth.
- Build/maintain local discovered-printer state.
- Keep LAN/Bambu Connect discovery as fallback when cloud is unavailable.
- Report inventory through existing control-plane pipeline.

### Local persistence
- `~/.printfarm-edge/bambu/discovered_printers.json`
- `~/.printfarm-edge/bambu/cache.json`

`discovered_printers.json` minimum fields:
- `identity` (`serial`/`device_id`)
- `serial`
- `device_id`
- `model`
- `name`
- `endpoint_hint`
- `source` (`cloud` or `lan_fallback`)
- `last_seen_at`

### Rules
- Primary source: cloud printer enumeration.
- Fallback source: Bambu Connect/LAN when cloud auth/API path is unavailable.
- Merge/dedupe key: `serial`, then `device_id`.
- No edge-side adoption command.

6. [CODE] cloud list client + merge/fallback + persistence + inventory report integration

### Exit criteria
- With valid cloud auth, Bambu local state is cloud-derived.
- With cloud unavailable, state falls back to LAN/Bambu Connect.
- Restart preserves discovered-printer state.
- Existing flags and existing non-Bambu behavior are unchanged.

## Phase 3 — Print Command (Start Print)

### Scope
- Route Bambu `print` action from existing desired-state reconciliation to cloud execution path.
- Support artifact download, cloud upload/submit, and print start.
- Keep control-plane contract unchanged.

### Print execution path

2. Download artifact via existing artifact mechanism.
3. Submit/upload/start via Bambu cloud adapter.
4. Push current-state updates via existing state-push loop.

## Phase 4 — Remaining Commands + Full Lifecycle

### Scope
- Implement pause/resume/cancel actions.
- Implement progress/telemetry updates and terminal state handling.
- Implement refresh/reconnect/recovery behavior.

### Commands and lifecycle
- Implement desired-state action mapping for Bambu:
  - `pause`
  - `resume`
  - `stop` (mapped to cancel)
- Maintain normalized state model parity:
  - printer: `idle`, `queued`, `printing`, `paused`, `error`
  - job: `pending`, `printing`, `completed`, `failed`, `canceled`

### Resilience
- Token refresh on expiry.
- Reconnect/status stream recovery after transient failures.
- Persistent auth failures mark Bambu path as auth-required/manual-intervention without crashing agent.

## Risks and Mitigations
- Cloud API instability: isolate via `cloud` interfaces + endpoint overrides.
- MFA friction in headless environments: support `--bambu-mfa-code-cmd`.
- Sidecar complexity (if used): strict health check, timeouts, typed error mapping.
- Refactor regressions: characterization tests and payload parity tests before behavior changes.
